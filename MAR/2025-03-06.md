### 개념 정리

|  | 정의 | 비고 |
| --- | --- | --- |
| Spring | 기존의 무겁고 복잡한 EJB(Enterprise JavaBeans) 모델을 대체할 수 있는 자바 플랫폼을 위한 유연한 애플리케이션 프레임워크 | IoC 컨테이너: 객체의 생성, 의존성 및 생명주기 관리를 컨테이너가 담당한다.<br>이를 통해 개발자는 핵심 비즈니스 로직에 집중할 수 있으며, 애플리케이션 구성 요소들 간의 결합도가 낮아진다.<br><br>DI(의존성 주입): 객체 간의 의존 관계를 설정 파일 또는 어노테이션으로 외부에서 주입함으로써 느슨한 결합(Loose Coupling)을 구현한다.<br>이로 인해 재사용성과 단위 테스트의 용이성이 크게 향상된다.<br><br>AOP(관점 지향 프로그래밍): 횡단 관심사(로깅, 보안, 트랜잭션 등)를 핵심 비즈니스 로직에서 분리해 모듈화한다.<br>이로 인해 코드의 유지보수성과 재사용성이 높아지고, 코드를 깔끔하게 유지할 수 있다.<br><br>POJO(Plain Old Java Object): 특정 인터페이스 구현이나 상속 없이 일반적인 자바 객체만으로 개발할 수 있다.<br>스프링의 POJO 철학은 “필요한 애노테이션만 붙이면 컨트롤러, 서비스, 레포지토리 등을 만들 수 있다”는 점으로, 유연한 설계를 가능하게 한다.<br><br>트랜잭션 관리: 선언적 트랜잭션 관리와 프로그래밍적 트랜잭션 관리를 모두 지원한다.<br>이를 통해 데이터베이스 작업 등 엔터프라이즈 환경의 복잡한 트랜잭션 요구사항에 유연하게 대응할 수 있다.<br><br>유연한 설정 방식: XML, 어노테이션, Java Config 등 다양한 설정 방식을 지원한다.<br>현대 실무에서는 주로 Java Config과 어노테이션 기반 설정 방식이 표준으로 자리 잡았다.<br><br>테스트 용이성: DI, POJO 기반 설계를 통해 단위 테스트와 통합 테스트를 쉽게 작성할 수 있으며, Mock 객체, MockMvc 등 테스트 지원 도구를 제공하여 테스트를 더욱 용이하게 만든다. |
| Spring Boot | 복잡한 Spring 설정을 자동화하는 프레임워크 | Spring에선 개발자가 XML이나 자바 설정 파일을 사용하여 직접 모든 빈(bean)과 관련 설정을 구성해야 하지만, Spring Boot에선 자동 설정과 스타터(starter) 의존성 관리를 제공하여, 최소한의 설정으로도 Spring 애플리케이션을 쉽게 시작할 수 있다.<br><br>전통적인 Spring 애플리케이션은 복잡한 설정과 배포 구성이 필요하며, 웹 서버(예: Tomcat)를 별도로 구성해야 한다. 하지만 Spring Boot는 내장 웹 서버(예: Tomcat, Jetty 등)를 제공하여, 별도의 서버 설정 없이 바로 실행 가능한 독립 실행형 애플리케이션을 만들 수 있다. |
| MVC 패턴 | 애플리케이션을 **모델(Model)**, **뷰(View)**, **컨트롤러(Controller)** 세 가지 역할로 분리하여 개발하는 소프트웨어 아키텍처 패턴 | 모델(Model): 애플리케이션의 데이터와 비즈니스 로직을 담당<br>데이터 관리: 데이터의 저장, 검색, 수정 등과 관련된 작업 수행<br>비즈니스 로직: 애플리케이션의 핵심 규칙과 처리 과정 구현<br>뷰나 컨트롤러와 독립적으로 작동하여, 데이터와 로직의 변경이 다른 구성 요소에 미치는 영향을 최소화한다.<br><br>뷰(View): 사용자에게 정보를 보여주는 역할<br>UI 구성: 모델로부터 전달받은 데이터를 기반으로 사용자 인터페이스를 구성하고 표시<br>프리젠테이션: 사용자에게 데이터를 시각적으로 표현하며, 입력폼, 목록, 차트 등 다양한 형태의 출력물 제공<br>데이터 처리나 비즈니스 로직과 분리되어, 화면에 출력되는 부분만 담당한다.<br><br>컨트롤러(Controller): 사용자 입력(HTTP 요청 등)을 처리하고, 모델과 뷰 사이의 상호 작용을 관리<br>입력 처리: 사용자의 요청(예: 버튼 클릭, 폼 제출 등)을 받아들여 적절한 작업을 결정<br>모델과의 상호작용: 사용자의 요청에 따라 모델의 데이터를 조회하거나 수정하는 작업 수행<br>모델의 결과를 받아서 적절한 뷰를 선택하고, 사용자에게 그 결과를 보여준다.<br><br>MVC 패턴의 장점은 다음과 같다.<br><br>관심사 분리: 모델, 뷰, 컨트롤러로 역할이 분리되어 코드 응집도를 높이고, 복잡성을 낮춘다.<br><br>코드 유지보수성 향상: UI가 변경될 경우 뷰만 수정, 비즈니스 로직 변경은 모델만 수정 등으로, 각 영역별 수정 범위가 제한되어 유지보수가 용이하다.<br><br>테스트 용이성: 모델, 뷰, 컨트롤러가 독립적으로 개발・테스트될 수 있어 단위/통합 테스트가 편리하다.<br><br>개발 효율성 증대: 프론트엔드(뷰)와 백엔드(컨트롤러, 모델) 팀이 역할 분담해 협업하기 용이하며, 확장성과 재사용성도 높다. |
| Servlet | 웹 애플리케이션의 서버 측에서 실행되는 Java 프로그램으로, 클라이언트(예: 웹 브라우저)로부터 들어오는 요청을 처리하고, 그에 따른 응답을 생성하여 전송하는 역할을 한다. | 효율적인 요청 처리: 멀티스레딩을 지원하여 동시에 여러 요청을 처리할 수 있다.<br><br>확장성: 웹 애플리케이션의 요구 사항에 맞게 쉽게 확장할 수 있다.<br><br>유연성: 다양한 웹 기술과 쉽게 통합되어, 동적 웹 페이지 및 웹 애플리케이션 개발에 유용하다.<br><br>표준화: Java EE 표준의 일부이므로, 다양한 서버 및 컨테이너에서 일관된 동작을 보장한다. |
| Spring MVC | 스프링 프레임워크에서 제공하는 웹 MVC 모듈 | Spring MVC는 크게 다음 **핵심 컴포넌트**가 협력하여 동작합니다.<br><br>DispatcherServlet(Front Controller): 모든 HTTP 요청을 가장 먼저 받아서 처리하는 프론트 컨트롤러 역할을 하며, 웹 요청 흐름을 중앙 집중적으로 관리한다.<br>스프링 부트 자동 설정으로 기본적으로 `/` 에 매핑되며, 공통 로직(예: 로깅, 인증 등)을 한 곳에서 처리할 수 있다.<br><br>HandlerMapping: DispatcherServlet이 받은 요청 URL 등을 분석해 어떤 컨트롤러(Handler)가 해당 요청을 처리할지 결정하고 매핑한다.<br>주요 구현체로 RequestMappingHandlerMapping이 가장 일반적으로 사용되며, @RequestMapping과 @GetMapping 등의 어노테이션 기반 매핑을 지원한다.<br><br>HandlerAdapter: HandlerMapping을 통해 찾은 컨트롤러를 실제로 실행할 수 있도록 어댑터 패턴으로 설계되어 있다.<br>주요 구현체는 RequestMappingHandlerAdapter로, @Controller나 @RestController 기반의 컨트롤러를 실행한다.<br><br>Controller: 사용자 요청을 분석하고, 필요한 비즈니스 로직(Model)을 호출한 후 결과를 모델에 담아 뷰로 전달하거나, REST API의 경우 JSON/XML 형태로 응답한다.<br>@Controller는 HTML 뷰 렌더링을 위한 컨트롤러이며, @RestController는 REST API를 위한 컨트롤러(기본적으로 @ResponseBody가 포함됨)이다.<br><br>Model: 컨트롤러에서 뷰로 전달할 데이터를 담는 컨테이너 역할을 하며, 종류는 다음과 같다.<br>Model: 키-값 쌍 기반 인터페이스<br>ModelMap: Model의 구현체로 Map 형태<br>ModelAndView: 뷰 이름과 모델 데이터를 함께 관리 (과거 XML 설정에서 주로 사용됨)<br><br>ViewResolver: 컨트롤러가 반환한 뷰 이름(예: "products/list")을 실제 템플릿 파일(예: templates/products/list.html)과 연결한다.<br>Spring Boot에서는 Thymeleaf, Mustache 등의 Starter를 추가하면 자동 설정(ThymeleafAutoConfiguration 등)이 적용되어 쉽게 구성할 수 있다.<br><br>View: 모델 데이터를 기반으로 실제 화면(HTML, JSP, Thymeleaf 등)을 렌더링하는 컴포넌트이다.<br><br>스프링 MVC 내부 구조는 큰 흐름에서 `DispatcherServlet → HandlerMapping → HandlerAdapter → Controller → ViewResolver → View` 순서로 이루어진다.<br>그러나 @RestController를 사용하는 경우에는 컨트롤러 메서드가 반환하는 값이 바로 HTTP 응답 본문으로 변환되므로, ViewResolver를 거치지 않고 `HttpMessageConverter` 를 통해 JSON이나 XML 등으로 직접 변환되어 클라이언트에 전송된다. 즉, REST API에서는 뷰(View)를 렌더링할 필요가 없으므로, View 관련 단계가 생략된다. |
| Spring WebFlux | 스프링 프레임워크에서 제공하는 리액티브 스트림 기반의 비동기 모듈 | Spring WebFlux와 MVC의 차이점은 다음과 같다.<br><br>Spring WebFlux는 Reactive Streams를 기반으로 한 리액티브 프로그래밍 모델을 채택하여 비동기적이고 논블로킹 방식으로 요청을 처리하며, Netty, Undertow와 같은 논블로킹 서버에서 실행된다.<br><br>반면, Spring MVC는 전통적인 동기 및 블로킹 모델을 사용하여 요청-응답 사이클을 처리하며, 주로 Tomcat과 같은 서블릿 컨테이너에서 동작한다.<br><br>이러한 차이로 인해, I/O 중심의 애플리케이션에서는 Spring WebFlux가 높은 성능을 발휘하지만, 동시성이 높은 상황에서는 Spring MVC가 블로킹으로 인해 성능이 떨어질 수 있다. 따라서 데이터가 많은 실시간 애플리케이션에는 Spring WebFlux가, 전통적인 요청-응답 애플리케이션에는 Spring MVC가 더 적합하다고 할 수 있다. |
| Interceptor | DispatcherServlet과 컨트롤러 사이에서 동작하며, 특정 요청에 대한 전/후 처리 로직을 제공하는 컴포넌트 | 사전 처리 (Pre-processing): 컨트롤러가 요청을 처리하기 전에, 인터셉터는 요청을 가로채서 추가적인 검증이나 전처리 작업을 수행할 수 있다.<br>ex) 사용자 인증이나 권한 체크 수행<br><br>사후 처리 (Post-processing): 컨트롤러가 요청을 처리한 후, 뷰가 렌더링되기 전에 응답 데이터를 수정하거나 추가 작업을 수행할 수 있다.<br>ex) 공통 응답 헤더를 추가하거나 로깅 수행<br><br>요청 완료 후 처리 (After-completion): 뷰 렌더링이 완료된 후, 요청 처리가 완전히 끝난 시점에 인터셉터는 후처리 작업을 수행할 수 있다.<br>ex) 리소스 정리나 추가 로깅 수행 |
| Filter | 클라이언트의 HTTP 요청이 서블릿(또는 DispatcherServlet)에 도달하기 전에 실행되는 컴포넌트 | 범용성: 모든 서블릿 요청에 대해 적용될 수 있으며, DispatcherServlet 이전 단계에서 실행된다.<br><br>요청 전/후 처리: 요청이 서블릿에 도달하기 전과 응답이 클라이언트로 반환되기 전에 사전 또는 사후 처리를 수행할 수 있다.<br><br>적용 범위: 보통 인증, 로깅, 인코딩 설정, 보안 필터링 등 전역적인 작업에 사용된다.<br><br>제한 사항: 서블릿 필터는 Spring MVC의 내부 요청 처리 흐름에 직접 관여하지 않으며, DispatcherServlet 이후의 세부 작업(핸들러 매핑, 컨트롤러 실행 등)에는 영향을 주지 않는다. |
| Bean | 스프링 IoC 컨테이너에 의해 생성되고 관리되는 객체 | 빈은 스프링 IoC 컨테이너에 의해 생성, 의존성 주입, 생명주기 관리 등 전반적인 관리를 받는다.<br><br>빈은 XML 설정, 어노테이션(@Controller, @Bean..), Java Config 등 다양한 방식으로 스프링 컨테이너에 정의되며, 독립적인 객체로 설계되어 재사용성이 높고, 애플리케이션의 다양한 곳에서 활용될 수 있다.<br><br>빈은 다른 빈과의 의존 관계를 가질 수 있으며, 스프링 컨테이너로부터 의존성을 주입받아 결합도를 낮춘다. |
| POJO | 특정 프레임워크나 규약에 종속되지 않는 순수한 자바 객체 | POJO와 Bean의 차이점은 다음과 같다.<br><br>제약 조건<br>POJO: 특별한 제약 없이 자유롭게 설계할 수 있다.<br>Java Bean: 정해진 규약(기본 생성자, 프로퍼티 접근자, 직렬화 등)을 반드시 따라야 한다.<br><br>용도<br>POJO: 간단한 객체나 비즈니스 로직을 위한 기본 데이터 객체로 사용된다.<br>Java Bean: 주로 엔터프라이즈 애플리케이션에서 재사용 가능한 컴포넌트 등으로 활용된다.<br><br>구조<br>POJO: 필요한 기능만을 담은 최소한의 구조를 가진다.<br>Java Bean: 데이터를 캡슐화하고, 외부에서 데이터를 읽거나 쓸 수 있도록 접근자 메서드를 제공하여 구조가 다소 표준화되어 있다. |
| Gradle | 자바 프로젝트를 비롯한 여러 언어와 플랫폼의 빌드를 효율적으로 관리할 수 있도록 설계된 오픈 소스 기반의 빌드 자동화 도구 | Groovy 기반의 도메인 특화 언어(DSL)를 사용한 빌드 자동화 시스템으로, 프로젝트 구성을 선언할 때 XML 파일을 사용하지 않는다.<br>태스크 의존성 그래프를 기반으로 작동하며, 변경된 태스크만 실행하여 성능을 향상시킨다. |

### 오늘의 회고
지금까지 스프링부트로 여러 프로젝트를 진행해오면서 깊게 고민해 본 적이 없는 여러 개념들에 대해 배우는 시간이었다. 오늘의 개념들을 나의 지난 프로젝트에서 사용했는지, 사용했다면 어떤 방식으로 사용했는지 보면서 들으니 수업이 더 재밌게 느껴지고, 이해가 쉬웠던 것 같다. 그래도 더 깊게 파고들며 공부하는 시간이 부족하다는 생각이 들어서 과제를 진행하면서 개념 정리를 혼자서라도 다시 해볼 생각이다.
